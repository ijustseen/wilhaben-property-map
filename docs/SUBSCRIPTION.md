# Free vs Plus — продуктовая спецификация

Уточняет, **чем Plus отличается от Free** и что именно гейтить.
Общая монетизация (B2B, реклама, воронка): см. [`MONETIZATION.md`](../MONETIZATION.md).
Тарифы в коде: [`lib/plans.ts`](../lib/plans.ts). Checkout: Lemon Squeezy.

Статус: **спека для дизайна и реализации**, не юридический офер. При расхождении с `/pricing` — править код/копирайт под этот документ.

---

## 1. Принцип разделения

| Слой | Free | Plus |
|------|------|------|
| Понять рынок | да | да |
| Собрать shortlist | да | да |
| Не сидеть на карте сутками | нет (ручной заход) | да (алерты) |
| Упаковать shortlist для других | нет | да (экспорт / share) |

**Правило:** если фича нужна, чтобы *увидеть и сравнить жильё*, она в Free.  
Если фича нужна, чтобы *охотиться неделями без постоянного открытия сайта*, она в Plus.

Plus продаёт **время и внимание**, не доступ к карте.

Типичный платящий сценарий: 4–10 недель вокруг старта семестра → подписка на 1–3 месяца → отмена. Это нормально и должно быть явно в копирайте («cancel anytime»).

## 3. Free — конкретный объём

### Входит

1. **Карта** всех `available` городов и университетов.
2. **Фильтры:** тип жилья, rent min/max, комнаты, PLZ (как сейчас).
3. **Маркеры + деталь объявления**, переход на источник (willhaben / wg-gesucht / dorm).
4. **Commute** от выбранного объявления к **одному** выбранному университету.
5. **Избранное** (логин): без жёсткого лимита на старте; синк между устройствами.
6. **Сердечко на карте** для сохранённых (уже есть) — без авто-уведомлений.

### Не входит (и не «почти Plus»)

- Письма / push о новых объявлениях.
- Письма об изменении цены или снятии объявления.
- Сохранение поискового профиля с `notify=true`.
- PDF / share-ссылка shortlist.
- Второй кампус в одном сравнении commute.

### Soft limits (только если появится abuse)

Не вводить на старте. Если нужно позже:

- Soft cap избранного (например 100) + баннер «наведи порядок / Plus для алертов», **не** жёсткий paywall на 3 сохранения.
- Rate-limit API избранного — защита инфраструктуры, не монетизация.

## 4. Plus — фичи с чёткими границами

### P1 — Saved search + email (главный дифференциатор)

**Зачем платят:** студент не может обновлять карту каждые 2 часа.

**Определение «saved search»:**

- `city_id` (обязательно)
- `university_id` (желательно — для deep link «Open on map»)
- `source` (apartments | shared | dorms)
- снимок фильтров: `priceFrom`, `priceTo`, `rooms[]`, `areaId` / PLZ
- `notify_email: true`

**Лимит:** **5** активных поисков на аккаунт (как в `PLANS`).

**Что шлётся:**

- Новое объявление, попавшее в выборку поиска с момента последней проверки.
- Не чаще одного письма на поиск за окно (см. ниже), digest предпочтительнее spam.

**Частота (MVP):**

- Проверка: 2–4 раза в сутки (cron / queue).
- Письмо: digest «N новых за период» + ссылки `mapFavoriteListingPath`-style на карту.
- Не real-time в MVP.

**Free-аналог без Plus:**

- Можно применить фильтры и оставить вкладку / bookmark URL карты — без серверного watcher и без email.

**Гейт:** создание `saved_searches` с `notify_email` и отправка писем — только `plan === 'plus'`.  
UI: кнопка «Notify me» → `/pricing` или checkout, если Free.

---

### P2 — Favourite watchdog (цена / статус)

**Зачем платят:** избранное уже есть → Plus делает его «живым».

**События MVP:**

1. **Price change** — `price` / `priceDisplay` изменился относительно snapshot при сохранении (или последнего известного).
2. **Gone** — объявление больше не резолвится / 404 / снято с выдачи (best-effort по источнику).

**Не в MVP:** «репост того же объекта», «дней на сайте», push.

**Скоуп:** только избранное пользователя, не весь рынок.  
**Частота:** та же cron-волна, что и для searches, или реже (1–2×/сутки).  
**Письмо:** «3 favourites changed» digest, не письмо на каждый pin.

**Гейт:** джоба алертов читает только users с `plan = plus`. Free видит статичный shortlist.

---

### P3 — PDF export shortlist

**Зачем платят:** показать родителям / соседям по WG / себе «короткий список».

**Содержимое PDF (MVP):**

- Заголовок + дата
- До N избранных (например все, или выбранные чекбоксами)
- На объявление: title, price, rooms/area если есть, address, source, URL, optional commute summary к текущему uni, monthly cost estimate если уже считается в UI

**Не в MVP:** красивый бренд-бук на 10 страниц, multi-language layout.

**Гейт:** `POST /api/export/favorites` (или аналог) проверяет Plus server-side.

**Альтернатива дешевле в разработке:** share-link (read-only страница shortlist) вместо PDF — тоже Plus; PDF можно вторым шагом. Для копирайта на `/pricing` сейчас обещан PDF — либо реализовать, либо поправить `lib/plans.ts` до релиза.

---

### P4 — Dual-campus commute

**Зачем платят:** узкий, но понятный кейс (два кампуса / партнёр / смена факультета).

**Поведение:**

- Free: commute только к `university` из URL/фильтров.
- Plus: в панели детали — выбрать **второй** университет (того же или другого города) и видеть оба времени рядом.

**Приоритет:** ниже P1–P2. Можно сдвинуть, если алерты ещё не живут. Не блокирует запуск Plus ради алертов.

---

### P5 — Priority support

Только строка на pricing. Канал: GitHub issue / email с чека Lemon.  
**Не** строить SLA и отдельную очередь ради этого пункта.

## 5. Что не продаём (антипаттерны)

| Идея | Почему нет |
|------|------------|
| Спрятать dorms / WG за Plus | Ломает доверие и SEO |
| Один город на Free | Ломает позиционирование Austria-wide |
| Commute целиком только в Plus | Убивает отличие от willhaben |
| Лимит «3 избранных» | Враждебно в стрессовом сценарии |
| «Свежее, чем у Free» как главный pitch | Недоказуемо без тяжёлой ops |
| Plus = только без рекламы | Ок как бонус, не как единственная ценность |

Платные partner pins на карте — отдельный B2B/рекламный трек (`MONETIZATION.md`), не фича Plus для студента.

## 6. Порядок реализации (рекомендуемый)

```
1. Lemon webhook → app_users.plan = free|plus   (уже частично есть)
2. Soft-upsell UI на Favourites + после Apply filters
3. saved_searches table + CRUD (Plus-only write)
4. Cron: match new listings → email digest          ← P1
5. Cron: re-check favourites price/gone → email     ← P2
6. Export PDF (или share link)                      ← P3
7. Dual-campus commute                              ← P4
```

**Не запускать оплату «вхолостую»:** пока нет P1 или P3, на `/pricing` честно «coming soon» или оставить checkout только после готового алертера.

Минимальный «честный» Plus launch: **webhook + P1** (даже без PDF).  
Идеальный сезон-launch: **P1 + P2**. PDF — nice-to-have в тот же месяц.

## 7. Где показывать upsell

| Место | Сообщение |
|-------|-----------|
| Favourites (пусто / есть список) | «Get email when these change — Plus» |
| После Apply filters | «Save this search & get alerts» |
| Listing detail, рядом с Save | вторично; Save остаётся Free |
| `/pricing` | полная таблица Free / Plus |

Не блокировать загрузку карты и API listings для Free.

## 8. Копирайт (тон)

- Free: «Pick your university. Find housing on the map.»
- Plus: «Get notified while you hunt — new matches and changes on saved flats.»
- Цена: «€4.90/mo · cancel anytime» (сезонная подписка, не «pro forever»).

Аудитория: студенты (часто international), UI продукта на английском; **язык алертов** — открытый вопрос (EN MVP, DE позже или по `locale` cookie).

---

## 9. Метрики до и после Plus

**До алертов (здоровье Free):**

1. % сессий карты с выбранным университетом  
2. % залогиненных с ≥1 favourite  
3. Return visit ≤7 дней после первого save  

**После Plus:**

4. Free → checkout start → paid (цель ориентир 2–5% в пик, см. `MONETIZATION.md`)  
5. Alert email open rate + клик «Open on map»  
6. Median длительности подписки (ожидаемо 1–3 месяца)  
7. Churn после «нашёл жильё» — не считать провалом продукта  

Если save rate низкий — сначала UX избранного, не алерт-движок.

## 10. Технические якоря

| Тема | Решение |
|------|---------|
| Источник plan | `app_users.plan` (`free` \| `plus`), пишет Lemon webhook |
| Проверка доступа | Только server-side (API export, создание saved search, cron) |
| Схема searches | `saved_searches` (см. набросок в `docs/SUPABASE.md`); для текущего auth — аналог `app_saved_searches` |
| Email | Провайдер TBD (Resend / Postmark / Lemon transactional — решить при P1) |
| Идемпотентность алертов | Хранить `last_seen_listing_ids` / hash snapshot цены, чтобы не дублировать письма |

Карта, фильтры, избранное CRUD — **без** проверки Plus.

## 11. Открытые решения (зафиксировать перед кодом P1)

1. **Digest vs near-real-time** — рекомендация: digest 2–4×/день.  
2. **Язык писем** — EN only в MVP?  
3. **PDF vs share-link первым** — PDF обещан на pricing; share проще, но нужен апдейт `lib/plans.ts`, если меняем promise.  
4. **Лимит 5 searches** — ок; удаление/пауза notify без удаления поиска?  
5. **Семестровый пакет** (3 месяца prepaid со скидкой) — не в v1.  
6. **Campus / ÖH seats** — B2B, не смешивать с B2C Plus в одном тарифе.

## 12. Definition of done для «Plus живой»

- [ ] Пользователь с `plan=plus` может создать ≤5 saved searches с email.  
- [ ] Free при том же UI видит upsell, запись в БД с notify не создаётся.  
- [ ] Cron хотя бы раз в сутки шлёт письмо при появлении матча (staging).  
- [ ] Отмена в Lemon → `plan=free` → новые алерты не уходят (grace: до конца оплаченного периода, если Lemon так отдаёт).  
- [ ] Карта и избранное у Free без регрессий.

Когда P1 закрыт — обновить этот файл статусом «shipped» у соответствующих пунктов и синхронизировать `/pricing` с фактическим поведением.
